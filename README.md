# 1. Объясните, каковы основные обязанности веб-разработчика?

Веб-разработчик — это специалист, который занимается созданием, развитием и поддержкой веб-приложений и веб-сайтов.

Основные обязанности веб-разработчика:

1. **Разработка пользовательского интерфейса (Frontend)**

Включает:

* создание структуры страниц;
* разработку интерактивных элементов;
* адаптацию интерфейса под разные устройства;
* работу с браузерными API.

Используемые технологии:

* HTML;
* CSS;
* JavaScript;
* TypeScript;
* React, Angular, Vue и другие фреймворки.

---

2. **Разработка серверной части (Backend)**

Включает:

* создание серверной логики;
* разработку API;
* обработку запросов пользователей;
* работу с базами данных;
* реализацию бизнес-логики.

Используемые технологии:

* Node.js;
* C#/.NET;
* Java;
* Python;
* PHP;
* базы данных SQL и NoSQL.

---

3. **Работа с базами данных**

Веб-разработчик должен уметь:

* проектировать структуру данных;
* создавать запросы;
* оптимизировать работу с данными;
* обеспечивать безопасность хранения информации.

Примеры:

* PostgreSQL;
* MySQL;
* MongoDB;
* Redis.

---

4. **Разработка и интеграция API**

Включает:

* создание REST API;
* работу с GraphQL;
* интеграцию сторонних сервисов;
* обработку ошибок и статусов HTTP.

---

5. **Обеспечение безопасности**

Веб-разработчик должен учитывать:

* защиту от XSS;
* защиту от CSRF;
* безопасную обработку пользовательских данных;
* аутентификацию и авторизацию;
* использование HTTPS.

---

6. **Оптимизация производительности**

Включает:

* уменьшение размера ресурсов;
* оптимизацию загрузки страниц;
* работу с кешированием;
* улучшение скорости выполнения JavaScript;
* оптимизацию запросов к серверу.

---

7. **Тестирование и поддержка**

Разработчик занимается:

* написанием unit-тестов;
* исправлением ошибок;
* анализом производительности;
* сопровождением приложения.

---

8. **Работа с инструментами разработки**

Например:

* Git;
* системы сборки (Webpack, Vite);
* CI/CD;
* Docker;
* системы управления задачами.

---

# 3. Что такое HTTP?

HTTP (**HyperText Transfer Protocol**) — это протокол передачи данных между клиентом и сервером.

Используется для обмена ресурсами:

* HTML;
* JSON;
* изображения;
* файлы;
* видео;
* другие данные.

HTTP работает по модели:

```
Client → Request → Server

Client ← Response ← Server
```

---

## HTTP-запрос состоит из:

### Метод

Определяет действие:

| Метод  | Назначение           |
| ------ | -------------------- |
| GET    | Получение данных     |
| POST   | Отправка данных      |
| PUT    | Полное обновление    |
| PATCH  | Частичное обновление |
| DELETE | Удаление             |

---

### URL

Адрес ресурса:

```
https://example.com/users
```

---

### Headers

Дополнительная информация:

Пример:

```http
Content-Type: application/json
Authorization: Bearer token
```

---

### Body

Передаваемые данные:

```json
{
    "name": "John"
}
```

---

## HTTP-ответ содержит:

### Status Code

Код результата:

| Код | Значение              |
| --- | --------------------- |
| 200 | Успешный запрос       |
| 201 | Создан ресурс         |
| 400 | Ошибка клиента        |
| 401 | Требуется авторизация |
| 403 | Доступ запрещен       |
| 404 | Не найдено            |
| 500 | Ошибка сервера        |

---

## Особенности HTTP

HTTP:

* является stateless-протоколом;
* не хранит состояние между запросами;
* работает поверх TCP;
* является текстовым протоколом.

---

# 4. Что такое HTTPS?

HTTPS (**HyperText Transfer Protocol Secure**) — это защищенная версия HTTP.

HTTPS использует HTTP поверх протокола шифрования TLS.

Схема:

```
HTTP + TLS = HTTPS
```

---

## Основные преимущества HTTPS:

### 1. Шифрование данных

Все данные между клиентом и сервером передаются в зашифрованном виде.

Защищает от:

* перехвата данных;
* прослушивания трафика.

---

### 2. Аутентификация сервера

HTTPS использует SSL/TLS-сертификаты.

Сертификат подтверждает:

* кому принадлежит сайт;
* что соединение установлено с правильным сервером.

---

### 3. Целостность данных

TLS гарантирует, что данные:

* не были изменены во время передачи;
* не были подменены злоумышленником.

---

## Как работает HTTPS?

Упрощенно:

1. Клиент подключается к серверу.
2. Сервер отправляет сертификат.
3. Клиент проверяет сертификат.
4. Создается защищенный канал.
5. Дальше передаются HTTP-данные.

---

# 5. Преимущества HTTP/2 по сравнению с HTTP/1.1

HTTP/2 — это новая версия HTTP, разработанная для повышения скорости передачи данных.

---

# 1. Multiplexing (мультиплексирование)

В HTTP/1.1 каждый запрос часто требовал отдельного соединения.

Например:

```
Request 1 → Connection 1
Request 2 → Connection 2
Request 3 → Connection 3
```

HTTP/2 позволяет отправлять несколько запросов через одно TCP-соединение:

```
Connection

 ├── Request 1
 ├── Request 2
 └── Request 3
```

Преимущество:

* меньше соединений;
* меньше задержек;
* быстрее загрузка страницы.

---

# 2. Binary Protocol

HTTP/1.1 использует текстовый формат.

Пример:

```http
GET /index.html HTTP/1.1
```

HTTP/2 использует бинарное представление.

Преимущества:

* быстрее обработка;
* меньше ошибок;
* эффективнее передача данных.

---

# 3. Header Compression (HPACK)

HTTP-заголовки часто повторяются:

Например:

```http
Cookie:
Authorization:
User-Agent:
```

HTTP/2 использует алгоритм HPACK для сжатия заголовков.

Преимущества:

* уменьшение размера запросов;
* экономия трафика.

---

# 4. Server Push

HTTP/2 позволяет серверу заранее отправить ресурсы, которые понадобятся клиенту.

Например:

Пользователь запросил:

```
index.html
```

Сервер сразу отправляет:

```
index.html
style.css
app.js
```

---

# 5. Приоритеты запросов

HTTP/2 позволяет задавать приоритеты ресурсов.

Например:

Сначала загрузить:

1. HTML;
2. критический CSS;
3. JavaScript;
4. изображения.

---

# 6. Лучшее использование TCP-соединений

HTTP/2 уменьшает количество соединений между клиентом и сервером.

Преимущества:

* меньше нагрузка на сервер;
* меньше задержки;
* лучше работа на мобильных сетях.

---

# Сравнение HTTP/1.1 и HTTP/2

| Возможность         | HTTP/1.1    | HTTP/2   |
| ------------------- | ----------- | -------- |
| Формат              | Текстовый   | Бинарный |
| Соединения          | Несколько   | Одно     |
| Multiplexing        | Нет         | Есть     |
| Сжатие заголовков   | Нет         | HPACK    |
| Server Push         | Нет         | Есть     |
| Приоритеты запросов | Ограниченно | Есть     |
| Производительность  | Ниже        | Выше     |

---

# 6. Перечислите несколько способов уменьшить время загрузки страницы?

Время загрузки страницы напрямую влияет на пользовательский опыт, SEO и производительность приложения.

Основные способы оптимизации:

---

## 1. Оптимизация изображений

Изображения часто являются самыми тяжелыми ресурсами страницы.

Способы оптимизации:

* использовать современные форматы:

  * WebP;
  * AVIF;
* уменьшать размер изображений;
* использовать адаптивные изображения;
* применять lazy loading.

Пример:

```html
<img 
    src="image.webp"
    loading="lazy"
    alt="Image"
/>
```

---

## 2. Минификация файлов

Удаление ненужных символов:

* пробелов;
* комментариев;
* переносов строк.

Минифицируются:

* JavaScript;
* CSS;
* HTML.

Пример:

До:

```javascript
function sum(a, b) {

    return a + b;

}
```

После:

```javascript
function sum(a,b){return a+b}
```

---

## 3. Сжатие ресурсов

Используются алгоритмы:

* Gzip;
* Brotli.

Сервер отправляет уменьшенный размер файлов.

Например:

```
app.js  →  2 MB
app.js.gz → 300 KB
```

---

## 4. Использование кеширования

Браузер может сохранять ресурсы локально.

Например:

* изображения;
* CSS;
* JavaScript;
* шрифты.

Настраивается через HTTP-заголовки:

```http
Cache-Control: max-age=31536000
```

---

## 5. Использование CDN

CDN (**Content Delivery Network**) хранит копии ресурсов на серверах по всему миру.

Преимущества:

* уменьшение задержки;
* быстрая загрузка статических файлов;
* снижение нагрузки на основной сервер.

---

## 6. Уменьшение количества HTTP-запросов

Каждый запрос имеет сетевые издержки.

Оптимизация:

* объединение файлов;
* использование спрайтов;
* удаление ненужных ресурсов;
* использование HTTP/2.

---

## 7. Code Splitting

Разделение JavaScript-кода на части.

Вместо загрузки всего приложения:

```
app.js (10 MB)
```

загружается только необходимый код:

```
main.js
profile.chunk.js
settings.chunk.js
```

Например, в React:

```javascript
const Profile = lazy(
    () => import("./Profile")
);
```

---

## 8. Lazy Loading

Загрузка ресурсов только при необходимости.

Применяется для:

* изображений;
* компонентов;
* видео;
* больших библиотек.

---

## 9. Оптимизация JavaScript

Включает:

* удаление неиспользуемого кода;
* уменьшение количества вычислений;
* использование debounce/throttle;
* оптимизацию рендера.

---

## 10. Оптимизация CSS

Используются:

* удаление неиспользуемых стилей;
* критический CSS;
* уменьшение количества файлов;
* правильный порядок загрузки.

---

## 11. Предзагрузка ресурсов

Используются:

```html
<link rel="preload">
```

Например:

```html
<link 
    rel="preload"
    href="font.woff2"
    as="font"
/>
```

---

## 12. Оптимизация сервера

Включает:

* уменьшение времени ответа API;
* оптимизацию запросов к базе данных;
* кеширование данных;
* использование балансировки нагрузки.

# 8. Какие виды уязвимостей существуют в веб-приложениях?

Основные уязвимости веб-приложений описывает стандарт OWASP.

---

# 1. XSS (Cross-Site Scripting)

Внедрение вредоносного JavaScript-кода на страницу.

Пример:

```html
<script>
alert("hack")
</script>
```

Защита:

* экранирование данных;
* Content Security Policy;
* валидация пользовательского ввода.

---

# 2. CSRF (Cross-Site Request Forgery)

Атака, при которой пользователь выполняет нежелательное действие через другой сайт.

Пример:

Пользователь авторизован в банке.

Злоумышленник отправляет скрытый запрос:

```
Перевести деньги
```

Защита:

* CSRF-токены;
* SameSite Cookies;
* проверка Origin.

---

# 3. SQL Injection

Внедрение SQL-кода через пользовательские данные.

Опасный пример:

```sql
SELECT * FROM users 
WHERE name = 'input'
```

Защита:

* параметризованные запросы;
* ORM;
* валидация данных.

---

# 4. Broken Authentication

Ошибки в системе авторизации:

* слабые пароли;
* неправильное хранение токенов;
* отсутствие MFA.

---

# 5. Утечки данных

Причины:

* неправильные права доступа;
* хранение паролей без хеширования;
* открытые API.

---

# 6. Insecure Deserialization

Небезопасная обработка сериализованных данных.

Может привести к:

* выполнению кода;
* изменению данных.

---

# 7. Clickjacking

Пользователь нажимает элементы скрытого интерфейса.

Защита:

```http
X-Frame-Options: DENY
```

---

# 8. SSRF (Server-Side Request Forgery)

Злоумышленник заставляет сервер выполнять запросы к внутренним ресурсам.

---

# 9. Path Traversal

Получение доступа к файлам вне разрешенной директории.

Пример:

```
../../config.json
```

---

# 10. DDoS

Атака большим количеством запросов.

Защита:

* rate limiting;
* CDN;
* фильтрация трафика.

---

# 9. Что такое клиент-серверная архитектура? Как она работает?

Клиент-серверная архитектура — это модель взаимодействия, где система разделена на две основные части:

* клиент;
* сервер.

---

# Клиент

Клиент — это программа, которая отправляет запросы.

Примеры:

* браузер;
* мобильное приложение;
* desktop-приложение.

Отвечает за:

* интерфейс;
* взаимодействие пользователя;
* отображение данных.

---

# Сервер

Сервер принимает запросы и выполняет обработку.

Отвечает за:

* бизнес-логику;
* работу с базой данных;
* авторизацию;
* хранение данных.

---

# Как происходит взаимодействие?

Схема:

```
Пользователь

     ↓

Клиент (Browser)

     ↓ HTTP Request

Сервер

     ↓

База данных

     ↑

HTTP Response

     ↑

Клиент
```

---

Пример:

Пользователь открывает профиль:

1. Браузер отправляет:

```http
GET /profile
```

2. Сервер:

* проверяет пользователя;
* получает данные из БД;
* формирует ответ.

3. Клиент получает:

```json
{
    "name": "Alex"
}
```

4. Интерфейс отображает данные.

---

# Преимущества архитектуры

* разделение ответственности;
* масштабируемость;
* удобство поддержки;
* возможность использовать разные клиенты;
* централизованное управление данными.

---

# 10. Что вы знаете о криптографическом шифровании в программировании?

Криптография — это область компьютерных наук, занимающаяся защитой информации с помощью математических алгоритмов.

Основные задачи:

* конфиденциальность;
* целостность;
* аутентификация;
* подтверждение авторства.

---

# Основные виды криптографии

## 1. Симметричное шифрование

Используется один ключ:

```
Один ключ:

Шифрование → Расшифрование
```

Примеры алгоритмов:

* AES;
* ChaCha20.

Преимущества:

* высокая скорость.

Недостаток:

* проблема безопасной передачи ключа.

---

# 2. Асимметричное шифрование

Используются два ключа:

* публичный;
* приватный.

Схема:

```
Public Key → Encryption

Private Key → Decryption
```

Примеры:

* RSA;
* ECC.

Используется в:

* HTTPS;
* цифровых подписях.

---

# 3. Хеширование

Хеширование превращает данные в уникальную строку фиксированной длины.

Примеры:

* SHA-256;
* SHA-3;
* bcrypt.

Используется для:

* хранения паролей;
* проверки целостности файлов.

---

Пример:

```
password

↓

SHA-256

↓

5e884898da...
```

---

# 4. Электронная цифровая подпись

Используется для подтверждения:

* автора данных;
* целостности сообщения.

---

# Криптография в веб-разработке

Применяется для:

* HTTPS/TLS;
* JWT-токенов;
* хранения паролей;
* защиты API;
* шифрования данных.

---

# Важные правила безопасности

Нельзя:

* хранить пароли в открытом виде;
* использовать устаревшие алгоритмы;
* хранить секретные ключи в коде.

Нужно:

* использовать HTTPS;
* применять современные алгоритмы;
* хранить секреты в переменных окружения;
* использовать безопасное управление ключами.

---

# 11. Что такое асинхронность, параллельность, синхронность, многопоточность?

Эти понятия описывают разные способы выполнения задач в программировании.

---

# Синхронность (Synchronous)

**Синхронное выполнение** — это последовательное выполнение операций, когда следующая операция начинается только после завершения предыдущей.

Пример:

```javascript
id="v9r0v3"
console.log("Start");

console.log("Process");

console.log("End");
```

Результат:

```text
Start
Process
End
```

Каждая операция блокирует выполнение следующей.

---

## Пример из жизни

Один человек готовит обед:

1. Нарезает продукты.
2. Готовит блюдо.
3. Моет посуду.

Следующая задача начинается после завершения предыдущей.

---

## Плюсы синхронного выполнения:

* простой код;
* легко понимать;
* удобно для последовательных операций.

## Минусы:

* блокировка выполнения;
* медленная работа при долгих операциях.

---

# Асинхронность (Asynchronous)

**Асинхронность** — это выполнение операций, при котором программа может продолжать работу, не ожидая завершения долгой задачи.

Например:

* запрос к серверу;
* чтение файла;
* таймер;
* работа с базой данных.

---

Пример JavaScript:

```javascript
id="9d0g8n"
console.log("Start");


setTimeout(() => {

    console.log("Async operation");

}, 2000);


console.log("End");
```

Результат:

```text
Start
End
Async operation
```

Таймер выполняется позже, а основной поток продолжает работу.

---

## Асинхронность в JavaScript

JavaScript использует:

* Event Loop;
* Callback;
* Promise;
* async/await.

Пример:

```javascript
id="h7x4rs"
async function getData(){

    const response =
        await fetch("/api");

    return response.json();

}
```

---

## Преимущества асинхронности:

* приложение не блокируется;
* эффективное использование ресурсов;
* удобная работа с сетевыми операциями.

---

# Параллельность (Parallelism)

**Параллельность** — это выполнение нескольких задач одновременно.

Для настоящей параллельности обычно требуется несколько вычислительных ядер.

Пример:

```
CPU Core 1:
Task A

CPU Core 2:
Task B
```

Обе задачи выполняются одновременно.

---

## Пример

Обработка большого изображения:

```
Поток 1 → обработка первой части изображения

Поток 2 → обработка второй части изображения
```

---

## Где используется:

* вычисления;
* обработка больших данных;
* машинное обучение;
* серверные приложения.

---

# Многопоточность (Multithreading)

**Многопоточность** — это выполнение нескольких потоков внутри одного процесса.

Поток (**thread**) — это отдельная последовательность выполнения инструкций.

---

Пример:

Одно приложение:

```
Process: Browser

 ├── Thread 1 → UI
 ├── Thread 2 → Network
 └── Thread 3 → Rendering
```

---

## Пример языков с многопоточностью:

* C# (`Thread`, `Task`);
* Java (`Thread`);
* C++;
* Rust.

---

## JavaScript и многопоточность

Основной JavaScript-поток является однопоточным.

Однако существуют:

* Web Workers;
* Worker Threads в Node.js.

Пример:

```javascript
id="zqk6ae"
const worker =
    new Worker("worker.js");
```

---

# Разница между понятиями

| Понятие         | Описание                                           |
| --------------- | -------------------------------------------------- |
| Синхронность    | Задачи выполняются последовательно                 |
| Асинхронность   | Задача запускается и результат приходит позже      |
| Параллельность  | Несколько задач выполняются одновременно           |
| Многопоточность | Несколько потоков выполняют задачи внутри процесса |

---

# 12. Что такое REST-архитектура?

REST (**Representational State Transfer**) — это архитектурный стиль построения веб-сервисов, основанный на взаимодействии клиента и сервера через HTTP.

REST используется для создания API.

---

Основная идея:

> Клиент обращается к ресурсам сервера через стандартные HTTP-методы.

---

# Основные принципы REST

## 1. Клиент-серверное разделение

Frontend и Backend являются независимыми.

Например:

```
React приложение

        ↓ HTTP

REST API

        ↓

Database
```

---

## 2. Stateless (отсутствие состояния)

Каждый запрос должен содержать всю необходимую информацию.

Сервер не хранит состояние клиента между запросами.

Пример:

```http
GET /users/10

Authorization: Bearer token
```

---

## 3. Работа с ресурсами

В REST всё является ресурсом:

Примеры:

```
Users

Products

Orders
```

Ресурс имеет URL:

```
/users
/products
/orders
```

---

## 4. Использование HTTP-методов

| Метод  | Действие             |
| ------ | -------------------- |
| GET    | Получение данных     |
| POST   | Создание данных      |
| PUT    | Полное обновление    |
| PATCH  | Частичное обновление |
| DELETE | Удаление             |

---

Пример API:

Получить пользователя:

```http
GET /api/users/5
```

Ответ:

```json
{
    "id":5,
    "name":"Alex"
}
```

---

Создать пользователя:

```http
POST /api/users
```

Body:

```json
{
    "name":"Alex"
}
```

---

Удалить пользователя:

```http
DELETE /api/users/5
```

---

# REST API особенности

Преимущества:

* простота;
* масштабируемость;
* использование стандартов HTTP;
* независимость клиента и сервера.

Недостатки:

* иногда требуется много запросов;
* нет встроенного управления состоянием;
* возможна избыточная передача данных.

---

# 13. HTTP запросы

HTTP-запрос — это сообщение от клиента к серверу с просьбой выполнить определенное действие.

---

Структура HTTP-запроса:

```
Request Line

Headers

Body
```

---

# HTTP методы

## GET

Получение данных.

Пример:

```http
GET /users
```

Ответ:

```json
[
    {
        "id":1,
        "name":"Alex"
    }
]
```

---

## POST

Создание нового ресурса.

Пример:

```http
POST /users
```

Body:

```json
{
    "name":"John"
}
```

---

## PUT

Полное обновление ресурса.

Пример:

```http
PUT /users/1
```

---

## PATCH

Частичное обновление.

Пример:

```http
PATCH /users/1
```

Body:

```json
{
    "name":"Bob"
}
```

---

## DELETE

Удаление ресурса.

Пример:

```http
DELETE /users/1
```

---

# HTTP Headers

Заголовки содержат дополнительную информацию.

Примеры:

Тип данных:

```http
Content-Type: application/json
```

Авторизация:

```http
Authorization: Bearer token
```

Кеширование:

```http
Cache-Control: max-age=3600
```

---

# HTTP Body

Тело запроса содержит данные.

Например:

```json
{
    "email":"test@mail.com",
    "password":"123456"
}
```

---

# 14. Группы ответов HTTP 2xx, 3xx, 4xx, 5xx

HTTP-коды ответа показывают результат выполнения запроса.

Они разделены на группы.

---

# 1xx — Informational

Информационные ответы.

Примеры:

| Код | Значение            |
| --- | ------------------- |
| 100 | Continue            |
| 101 | Switching Protocols |

Используются редко.

---

# 2xx — Success (Успешные ответы)

Запрос успешно выполнен.

---

## 200 OK

Стандартный успешный ответ.

Пример:

```http
GET /users

200 OK
```

---

## 201 Created

Ресурс успешно создан.

Пример:

```http
POST /users

201 Created
```

---

## 204 No Content

Успех, но без данных в ответе.

Например:

Удаление:

```http
DELETE /users/5

204 No Content
```

---

# 3xx — Redirection (Перенаправление)

Требуется дополнительное действие.

---

## 301 Moved Permanently

Постоянное перенаправление.

Пример:

```
old-site.com

↓

new-site.com
```

---

## 302 Found

Временное перенаправление.

---

## 304 Not Modified

Ресурс не изменился.

Используется при кешировании.

---

# 4xx — Client Errors (Ошибки клиента)

Ошибка в запросе клиента.

---

## 400 Bad Request

Некорректный запрос.

Например:

```json
{
    "email":"wrong format"
}
```

---

## 401 Unauthorized

Пользователь не авторизован.

Например:

* отсутствует токен;
* неверный токен.

---

## 403 Forbidden

Доступ запрещен.

Пользователь известен, но нет прав.

---

## 404 Not Found

Ресурс не найден.

Пример:

```http
GET /users/999999
```

---

## 409 Conflict

Конфликт данных.

Например:

* пользователь уже существует.

---

## 429 Too Many Requests

Слишком много запросов.

Используется при rate limiting.

---

# 5xx — Server Errors (Ошибки сервера)

Ошибка произошла на стороне сервера.

---

## 500 Internal Server Error

Общая ошибка сервера.

---

## 501 Not Implemented

Функция не реализована.

---

## 502 Bad Gateway

Проблема между серверами.

---

## 503 Service Unavailable

Сервис временно недоступен.

Причины:

* перегрузка;
* технические работы.

---

## 504 Gateway Timeout

Сервер не дождался ответа от другого сервера.

---

# Таблица HTTP-кодов

| ------ | ------------------- |
| 1xx    | Информация          |
| 2xx    | Успешное выполнение |
| 3xx    | Перенаправление     |
| 4xx    | Ошибка клиента      |
| 5xx    | Ошибка сервера      |

# Краткий ответ для собеседования

**Обязанности веб-разработчика:**

> Создание клиентской и серверной частей веб-приложений, работа с API и базами данных, обеспечение безопасности, тестирование и оптимизация производительности.

**Что должен знать веб-разработчик:**

> HTML, CSS, JavaScript, TypeScript, HTTP/HTTPS, браузерные API, фреймворки, базы данных, Git, инструменты сборки и основы безопасности.

**HTTP:**

> HTTP — протокол передачи данных между клиентом и сервером, основанный на модели запрос-ответ.

**HTTPS:**

> HTTPS — защищенная версия HTTP, использующая TLS для шифрования, аутентификации и защиты целостности данных.

**HTTP/2 преимущества:**

> HTTP/2 добавляет бинарный протокол, мультиплексирование запросов, сжатие заголовков, приоритеты запросов и Server Push, что повышает скорость загрузки веб-приложений.

**Уменьшение загрузки страницы:**

> Оптимизация изображений, минификация файлов, кеширование, CDN, lazy loading, code splitting, оптимизация JavaScript/CSS и уменьшение количества HTTP-запросов.

**Инструменты поиска багов:**

> Browser DevTools, React DevTools, логирование, Sentry, линтеры, тесты, Git и инструменты мониторинга.

**Уязвимости веб-приложений:**

> XSS, CSRF, SQL Injection, SSRF, утечки данных, проблемы авторизации, Clickjacking и DDoS.

**Клиент-серверная архитектура:**

> Модель, где клиент отправляет запросы, а сервер обрабатывает их и возвращает ответы. Клиент отвечает за интерфейс, сервер — за бизнес-логику и данные.

**Криптография:**

> Набор методов защиты информации с помощью шифрования, хеширования и цифровых подписей. Используется для защиты данных, авторизации и безопасной передачи информации.

**Асинхронность:**

> Возможность выполнять операции без блокировки основного потока, например сетевые запросы или таймеры.

**Параллельность:**

> Одновременное выполнение нескольких задач, обычно за счет нескольких вычислительных ядер.

**Многопоточность:**

> Выполнение нескольких потоков внутри одного процесса.

**REST:**

> Архитектурный стиль создания API, основанный на ресурсах, HTTP-методах и stateless-взаимодействии клиента и сервера.

**HTTP-запросы:**

> Сообщения клиента серверу, содержащие метод, URL, заголовки и тело запроса.

**HTTP-коды:**

> 2xx — успех, 3xx — перенаправление, 4xx — ошибка клиента, 5xx — ошибка сервера.
